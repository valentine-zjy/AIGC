# Story 1.3: 任务创建落库与匿名访问上下文

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 在上传被受理后获得一个私有的任务上下文来继续处理文档,
so that 我可以在无需传统注册流程的前提下安全访问自己的任务。

## Acceptance Criteria

1. 用户上传了一个通过校验的文件并完成任务创建后，系统保存任务基础记录、将原始文件写入私有对象存储，并将结构化任务元数据写入 PostgreSQL。
2. 任务创建成功后，用户后续继续访问当前任务时，系统通过匿名任务会话与任务级访问令牌校验其访问权限，且只允许在既定访问边界内读取该任务相关数据。
3. 上传接口涉及 cookie 绑定的变更操作时，系统必须执行 CSRF 防护，同时保证上传链路的传输与静态存储满足加密保护要求。

## Dev Notes

### Story Intent & Boundaries

- 本故事的目标是把 Story 1.2 的“上传受理成功”从前端可见占位态升级成真实的任务创建、持久化和受控访问模型。
- 本故事必须完成三件基础设施级工作：原始文件进入私有对象存储、任务元数据进入 PostgreSQL、匿名访问上下文收敛到任务级授权边界。
- 不要在本故事提前实现 BullMQ 入队、worker 消费、短轮询、处理状态轨道完整 UI、风险结果页、改写建议、导出或删除工作流；这些由 Story 1.4 与后续 Epic 承接。

### Previous Story Intelligence

- Story 1.2 已经把 `/api/uploads`、`UploadForm`、签名 `TASK_CONTEXT_COOKIE` 和 `tasks/[taskId]` 占位页接起来了，Story 1.3 必须建立在这些入口之上，而不是重做第二套上传链路。
- 当前 `apps/web/src/features/upload/upload-service.ts` 只在内存里生成 `task_${uuid}`，没有数据库、对象存储、真实访问边界，也没有统一 API response envelope；这正是本故事要替换的核心占位。
- 当前 `apps/web/src/app/(workspace)/tasks/[taskId]/page.tsx` 会读取 cookie，但仍把页面渲染成固定 `accepted` 文案；Story 1.3 要把它改成读取真实任务事实的最小任务页。
- 当前测试基线依旧是 `apps/web/src/app/task-context-check.mjs` 风格的最小回归校验；如在本故事引入更多 shared package，请在不破坏现有验证入口的前提下扩展。
- 当前目录仍不是 git 仓库；不要把“提交历史”“已有 branch 规范”作为实现前提。

### Developer Guardrails

- 严格遵循架构边界：`app/` 只放路由入口、页面壳层与 Route Handlers；业务编排放 `features/`；数据库、对象存储、契约放 `packages/*`。
- 不要把 Cloudflare R2 SDK、Drizzle 查询、cookie 授权校验和 API 响应拼装全部塞进一个超长 `apps/web/src/app/api/uploads/route.ts` 文件里。
- 不要继续让 `tasks/[taskId]` 页面只依赖 URL 参数和 bootstrap cookie 推断任务存在；任务存在性必须来自数据库事实。
- 不要因为 Story 1.3 需要真实持久化，就顺手把队列、worker、诊断流水线一起实现；闭环只到“上传 -> 任务创建成功 -> 任务页可安全重进”。
- 不要把私有原始文件写入 `public/`、本地仓库目录或任何默认公开可读位置。

### Architecture Compliance

- 数据与存储
  - 结构化任务元数据进入 `Managed PostgreSQL 17.x`。
  - 数据访问与迁移使用 `Drizzle ORM 0.44.5` + `drizzle-kit 0.31.4`。
  - 原始文件进入 `Cloudflare R2` 私有桶，不落数据库二进制列，不走公开桶。

- Web / BFF 边界
  - Web 仍采用 Next.js App Router + Route Handlers。
  - 正式接口遵循统一响应信封，不再继续扩散 Story 1.2 的裸 payload 返回风格。
  - 所有输入输出边界通过 Zod schema 定义，并在 `packages/contracts` 复用。

- 认证与授权
  - MVP 继续采用“匿名任务会话 + HttpOnly cookie + task-scoped opaque token”。
  - 授权粒度以任务为中心，而不是账号角色为中心。
  - 所有 cookie-bound mutation 都必须具备 CSRF 防护。

- 状态与后续衔接
  - Story 1.3 需要建立标准 task stage / status 初值，供 Story 1.4 的轮询和状态轨道直接复用。
  - Story 1.3 只需保证任务页能恢复最近一次已知状态，不负责完整进度可视化。

### Library / Framework Requirements

- Next.js
  - 继续使用 App Router、`route.ts`、`next/headers` cookie 能力与服务端任务页读取模式。
  - 上传与任务详情接口继续走 Route Handlers，不引入 GraphQL / tRPC。

- Drizzle
  - 通过 schema 文件 + `drizzle-kit generate` / `drizzle-kit migrate` 管理迁移。
  - schema 命名遵循架构约定，数据库列使用 `snake_case`，TypeScript/API 保持 `camelCase` / `PascalCase`。

- Cloudflare R2
  - 使用 S3 兼容方式接入；当前故事应封装为 `packages/storage` 内部实现，不让 Web 层直接依赖具体 SDK 细节。
  - 桶默认保持私有；后续如需下载，另走签名 URL 或受控下载接口，不在本故事开放。

- Zod
  - 任务创建输入、任务详情输出、授权失败输出、上传成功输出都要收敛到共享 schema。
  - 如当前 `uploadAcceptedResponseSchema` 与统一响应信封冲突，本故事应完成兼容迁移，而不是继续扩大例外。

### Latest Technical Information

- Next.js 官方文档当前明确将 `cookies()` 作为异步函数提供给 App Router，并允许在 Route Handlers 中读取/写入 cookie；这意味着 Story 1.3 可以在服务端页面与 Route Handlers 中统一收敛任务级 cookie / token 处理，而不必退回客户端会话方案。
- Next.js Route Handlers 官方示例继续基于 Web `Request` / `Response` API 与 `request.formData()` 处理表单请求；当前上传入口沿用 Route Handler 是正确方向，Story 1.3 只需把其内部实现从占位受理升级为真实持久化。
- Drizzle 官方 PostgreSQL 入门文档当前仍以 `drizzle-orm` + `pg` / `node-postgres` 作为标准接入路径；`drizzle-kit generate` 负责从 schema 生成 SQL migration，`drizzle-kit migrate` 负责按迁移目录执行到数据库，这与本项目要求的 schema + migration 双落地完全一致。
- Cloudflare R2 官方文档说明 bucket 默认不是公开可访的；同时官方 S3 兼容示例仍建议通过 `@aws-sdk/client-s3` 的 `S3Client` 访问 R2，region 使用 `auto`，endpoint 指向账户级 `r2.cloudflarestorage.com`。对本故事而言，这意味着应默认走私有对象写入，不暴露公开 bucket URL。
- Cloudflare R2 官方还提供 presigned URL 能力，但那更适合后续受控下载/导出故事；本故事只需要原始上传文件的私有写入和服务端持久化，不需要提前开放签名下载。
