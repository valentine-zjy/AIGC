# Story 2.1: 扫描任务编排与阶段状态写入

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 在文档上传后自动进入扫描流程并持续获得可理解的处理状态,
so that 我可以知道系统正在推进任务而不是陷入黑箱等待。

## Acceptance Criteria

1. **Given** 用户已成功上传一个受支持的文档并创建任务  
   **When** 系统完成任务受理  
   **Then** 后端将该任务投递到异步扫描流水线  
   **And** 扫描链路基于 BullMQ 与兼容 Redis 的队列系统实现
2. **Given** 扫描任务已进入处理队列  
   **When** Worker 开始执行扫描  
   **Then** 系统按上传、解析、扫描等阶段更新任务状态  
   **And** 每个阶段都具备可区分的状态值与时间戳，以支持用户反馈和故障定位
3. **Given** 扫描过程中系统负载升高或处理时间超出目标  
   **When** 用户等待结果  
   **Then** 系统仍需保持任务受理、排队和状态可见  
   **And** 通过清晰的排队中、处理中或延迟提示管理用户预期，而不是进入无反馈等待
4. **Given** 扫描过程中某一阶段失败  
   **When** 系统记录失败结果  
   **Then** 系统为任务写入明确的失败阶段、失败类型和基础处理记录  
   **And** 这些信息既可用于前台失败提示，也可供后续运营排障使用

## Dev Notes

### Story Intent & Boundaries

- Story 2.1 的目标不是产出 Top 风险段落本身，而是把“上传后进入扫描任务 -> worker 推进 -> 状态回写 -> 前台可见”这条异步链路做实。
- 当前仓库已经有任务状态轨道、queue package、worker 入口和占位阶段推进；本故事要在这些已有骨架上补齐真实编排与可验证状态，而不是重新发明第二套任务系统。
- Story 2.1 不负责实现风险结果列表、段落诊断、局部改写建议、一键优化或导出落地。这些分别属于后续 Story 2.2+ 与 Epic 3/4。
- Story 2.1 也不负责引入完整运营后台、死信平台或复杂调度系统；只需保证前台状态可信、worker 可运行、失败信息可追踪。

### Previous Story Intelligence

- Story 1.3 已建立匿名任务会话、task-scoped access token、对象存储写入与任务落库链路。Story 2.1 必须延续这条链路，而不是绕开现有任务模型新造任务表或授权模型。
- Story 1.4 已建立任务详情页、状态轨道、短轮询与失败反馈。Story 2.1 要做的是让这些状态从“真实 worker 写回”读取，而不是只依赖占位文案。
- Story 1.6 刚刚补齐了删除与保留期清理，说明当前任务模型、cookie 上下文与任务详情页已经继续演进；Story 2.1 的实现不能破坏已删除/已过期访问边界。
- 当前 memory queue / repository 已迁移到共享 temp runtime 路径，目的就是让 Web 和 Worker 在本地开发下共享状态，而不是依赖进程内 Map / 数组。Story 2.1 不要把这条修复回退掉。

### Technical Requirements

- 异步编排基线必须与架构保持一致：生产路径以 `BullMQ 5.58.5 + Redis-compatible queue` 为主；没有 `REDIS_URL` 的本地/测试环境才允许使用共享 runtime 的 memory 驱动。
- 阶段状态至少要覆盖 `upload -> parse -> scan` 的真实推进，并为每次状态变化写入时间戳；如果暂时保留 `rewrite` / `export` 阶段占位，必须保证不会向用户暗示这些能力已经在本故事中完成。
- 失败状态必须保留失败阶段、失败类型、失败说明、是否可重试与下一步提示，供前台与后续 ops 复用。
- 当处理时间超出常规预期时，系统必须优先显示“排队中 / 处理中 / 延迟”而不是静默等待，这直接对应 NFR4、NFR6、NFR16、NFR30。
- 前台继续采用短轮询即可；本故事不要擅自引入 SSE / WebSocket。

### Developer Guardrails

- 不要把 BullMQ 当成“将来再说”的注释位。只要环境提供 `REDIS_URL`，队列驱动就应走真实 BullMQ。
- 不要回退到 Web 进程内消费或 in-memory 数组队列；那会再次引入跨进程不可见、重复消费或开发环境行为失真问题。
- 不要为了 Story 2.1 提前实现 Top 风险结果、段落表、改写建议或导出产物，这会把故事边界打散。
- 不要把阶段状态和失败提示写死在前端。服务端事实状态必须来自任务记录与 worker 写回。
- 不要吞掉 worker 失败原因，也不要把 parse / scan / worker_unavailable / unexpected_worker_error 折叠成同一条模糊文案。

### Architecture Compliance

- 服务边界
  - `packages/queue` 是唯一入队/取队边界。
  - `apps/worker` 只消费队列和回写结果，不暴露公网 HTTP API。
  - `packages/db` 仍是唯一任务状态持久化边界。
- 数据边界
  - 原始文件继续放对象存储，结构化任务状态继续放 `processing_tasks` 或当前任务仓库实现。
  - Redis 只承载 queue / rate limit / 短期幂等窗口，不作为事实数据源。
- Web / UI 边界
  - `apps/web/src/app/api/uploads/route.ts` 与 `upload-service.ts` 只负责受理、入队与返回任务上下文。
  - `apps/web/src/features/tasks/*` 只负责读取和展示任务事实状态，不直接模拟 worker 推进。

### Library / Framework Requirements

- 继续沿用当前锁定版本：`next 16.2.1`、`react 19.2.4`、`@tanstack/react-query 5.87.4`、`zod 4.1.5`、`drizzle-orm 0.44.5`、`BullMQ 5.58.5`。
- 当前仓库测试风格仍以 `node --experimental-strip-types` 直跑 `.ts/.mjs` 文件为主；优先沿用现有方式，不在 Story 2.1 顺手引入新的测试框架。
- 前端继续沿用短轮询状态读取模式；不要因为 Story 2.1 额外引入新的状态管理库。
