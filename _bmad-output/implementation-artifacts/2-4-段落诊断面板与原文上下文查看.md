# Story 2.4: 段落诊断面板与原文上下文查看

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 在工作台中查看选中高风险段落的原文上下文、问题类型、判定原因和置信提示,
so that 我可以理解为什么它被标记，并决定是否优先处理该段落。

## Acceptance Criteria

1. **Given** 用户已进入扫描结果工作台并选中一个高风险段落  
   **When** 系统加载对应诊断信息  
   **Then** 页面在“段落诊断面板”中展示该段落的原文内容与必要上下文  
   **And** 用户能够明确识别当前正在处理的是哪一段
2. **Given** 用户查看某个高风险段落的诊断结果  
   **When** 面板渲染分析信息  
   **Then** 系统展示该段落对应的问题类型、风险等级和判定原因解释  
   **And** 解释内容应帮助用户理解其为何被判定为模板化、AI 味重或不像真实作者写法
3. **Given** 系统对某段结果具有不同确定性  
   **When** 用户查看该段落诊断信息  
   **Then** 面板展示高置信问题与可选优化项的区分提示  
   **And** 用户可以据此决定是否优先处理当前段落
4. **Given** 用户在高风险列表中切换选中项  
   **When** 当前段落发生变化  
   **Then** 诊断面板快速更新为新段落的原文、解释和状态信息  
   **And** 常规切换操作保持连续流畅，不出现明显卡顿或整页阻塞
5. **Given** 当前段落没有完整解释信息、上下文受限或结果暂未就绪  
   **When** 面板需要渲染对应状态  
   **Then** 系统展示清晰的占位、说明或空态反馈  
   **And** 告知用户当前缺少什么信息以及接下来会发生什么
6. **Given** 用户通过键盘或辅助技术使用诊断面板  
   **When** 用户在原文区域、解释区域和状态提示之间导航  
   **Then** 面板内容具备合理的语义结构、可见焦点和屏幕阅读器可理解文本  
   **And** 关键风险标签、处理状态和解释信息不只靠颜色表达

## Dev Notes

### Story Intent & Boundaries

- Story 2.4 的重点是把 Story 2.3 中的“进入诊断面板”入口落成真正可用的诊断主内容区，让用户看到原文、上下文、原因解释与置信提示。
- Story 2.4 不负责生成 2-3 个局部改写建议，也不负责接受 / 拒绝 / 回退决策；这些属于 Epic 3。
- Story 2.4 也不负责 PDF 扫描结果的专用只读边界提示；那属于 Story 2.5。

### Previous Story Intelligence

- Story 2.3 已完成高风险列表、筛选排序栏、诊断入口按钮、TanStack Query + Zustand 分工和桌面多面板骨架。
- 当前 `task-risk-workbench.tsx` 已有右侧诊断入口容器，但其中内容仍是 Story 2.3 占位文案，需要升级为真实诊断面板。
- 推荐排序规则已经在 Story 2.3 收口为“高风险优先，再按待处理优先，再按分值排序”；Story 2.4 不要破坏这一规则。

### Technical Requirements

- 继续沿用 `GET /api/tasks/[taskId]/risks` 作为风险列表入口，并新增单段诊断读取边界；优先与架构中已声明的 `/api/tasks/[taskId]/segments` 保持一致。
- 服务端事实数据继续进入 TanStack Query；Zustand 仍只保存选中段落、panel mode 等 UI 状态，不复制诊断结果整份到 store。
- 诊断数据至少应覆盖：原文正文、必要前文/后文、问题类型、风险等级、判定原因、置信提示、可选优化提示或状态说明。
- 当数据暂缺、上下文不足或任务未就绪时，接口应返回结构化状态，避免前端通过空字符串和 null 猜业务语义。

### Architecture Compliance

- 架构已明确工作台核心组件包含 `risk list`、`diagnosis panel`、`rewrite comparator`、`decision bar`；本故事只补 `diagnosis panel`。
- 架构目录已经预留 `apps/web/src/app/api/tasks/[taskId]/segments/route.ts` 与 `document_segments`；优先沿这一边界实现，而不是把段落上下文塞回 `risk_findings` 或 `processing_tasks`。
- 架构强调 Query 管服务端事实，Zustand 管本地交互状态；本故事必须延续这一分层。

### Library / Framework Requirements

- 延续当前锁定版本：`next 16.2.1`、`react 19.2.4`、`@tanstack/react-query 5.87.4`、`zustand 5.0.10`、`zod 4.1.5`、`drizzle-orm 0.44.5`。
- 优先继续使用纯函数 view model / service 形式，使诊断数据编排可通过 `node --experimental-strip-types` + `assert` 轻量验证。
