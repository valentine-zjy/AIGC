# Story 3.1: 段落改写建议生成与比较器

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 针对单个高风险段落查看 2-3 个局部改写建议，并与原文逐项对照,
so that 我可以在不重写整篇文档的前提下评估哪种改写更适合当前段落。

## Acceptance Criteria

1. **Given** 用户已进入工作台并选中一个高风险段落  
   **When** 系统完成该段落的建议生成  
   **Then** 页面展示该段落的 2-3 个局部改写建议  
   **And** 每个建议都与当前选中段落明确绑定，不影响其他段落
2. **Given** 用户查看某个高风险段落的建议区域  
   **When** “改写建议比较器”加载完成  
   **Then** 系统提供原文与多个候选建议的逐项对照视图  
   **And** 用户可以清楚识别原文、候选版本及其差异
3. **Given** 当前段落存在不同性质的问题  
   **When** 系统展示建议内容  
   **Then** 每个建议应与该段落的问题类型和判定原因相匹配  
   **And** 建议应服务于局部精修，而不是要求替换整篇文档
4. **Given** 某个段落的建议尚未生成完成、生成失败或当前无可用建议  
   **When** 用户查看建议区域  
   **Then** 系统展示明确的加载中、失败或空态反馈  
   **And** 告知用户当前状态及下一步预期，而不是显示不可理解的空白区域
5. **Given** 用户在多个高风险段落之间切换  
   **When** 当前选中段落变化  
   **Then** 比较器快速切换到对应段落的原文与候选建议  
   **And** 常规切换过程保持连续流畅，不出现明显卡顿或状态错乱
6. **Given** 用户通过键盘或辅助技术使用建议比较器  
   **When** 用户在原文、候选建议和辅助说明之间导航  
   **Then** 所有内容都具备清晰语义结构、可见焦点和可读文本说明  
   **And** 风险标签、状态提示和比较结果不只依赖颜色表达

## Dev Notes

### Story Intent & Boundaries

- Story 3.1 的重点是把 Epic 2 已完成的“风险识别 + 诊断解释”延伸为真正可比较的局部改写建议区，让用户开始进入精修闭环。
- Story 3.1 只负责建议生成、读取和比较展示，不负责采纳、拒绝、忽略、误判、回退或 `edit_operations` 写入；这些属于 Story 3.2 及之后的故事。
- Story 3.1 不负责一键优化、导出或 PDF 精确回写，仍需维持当前 MVP 边界。

### Previous Story Intelligence

- Story 2.3 已完成高风险列表、筛选排序栏、Zustand 选中状态与桌面工作台骨架。
- Story 2.4 已完成段落诊断面板、原文上下文、判定原因与置信提示读取链路。
- Story 2.5 已补齐 PDF 只读扫描边界提示，因此 Story 3.1 的比较器必须在 PDF 场景下保持“可查看建议，不承诺复杂版式原位回写”的表述边界。
- 当前仓库尚未存在 `rewrites` feature、`rewrite_options` schema 或建议读取接口，说明 Story 3.1 需要从数据契约到 UI 一次打通最小闭环。

### Technical Requirements

- 继续沿用任务子资源接口风格，建议新增 `GET /api/tasks/[taskId]/rewrite-options?paragraphId=...`，只读取当前选中段落的候选建议。
- 服务端事实数据继续进入 TanStack Query；Zustand 仍只保存 `selectedParagraphId`、比较器面板状态等 UI 状态，不复制整份候选建议到 store。
- 建议数据至少应覆盖：`paragraphId`、建议 ID、展示顺序、候选正文、策略说明、差异摘要、推荐提示与结构化状态。
- 建议内容必须与当前段落的 `issueType`、`diagnosisReason` 绑定，避免出现与问题类型无关的通用润色句。
- 当建议尚未就绪、生成失败或当前段落无可用建议时，接口应返回结构化状态，避免前端通过空数组和 null 猜业务语义。

### Architecture Compliance

- 架构已明确工作台核心组件包含 `risk list`、`diagnosis panel`、`rewrite comparator`、`decision bar`；本故事只补 `rewrite comparator`。
- 架构目录已经预留 `apps/web/src/app/api/tasks/[taskId]/rewrite-options/`、`apps/web/src/features/rewrites/` 与 `packages/db/src/schema/rewrite-options.ts`；优先沿这一边界实现，而不是把建议字段塞进 `risk_findings` 或 `document_segments`。
- 架构强调所有编辑修改最终通过 append-only `edit_operations` 表达，但 Story 3.1 只建立建议读取事实源，不提前落盘编辑决策。
- Worker 流水线应保持“segments / risks / rewrite options”分别建模，避免前端自己拼装候选建议。

### Library / Framework Requirements

- 延续当前锁定版本：`next 16.2.1`、`react 19.2.4`、`@tanstack/react-query 5.87.4`、`zustand 5.0.10`、`zod 4.1.5`、`drizzle-orm 0.44.5`。
