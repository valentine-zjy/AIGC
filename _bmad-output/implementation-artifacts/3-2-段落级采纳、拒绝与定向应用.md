# Story 3.2: 段落级采纳、拒绝与定向应用

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 对单个段落的候选建议执行采纳、拒绝，并且只把修改应用到当前选定段落,
so that 我可以精确控制哪些内容被改动，而不是替换整篇文档。

## Acceptance Criteria

1. **Given** 用户正在查看某个高风险段落的候选建议
   **When** 用户选择其中一个建议并执行采纳
   **Then** 系统仅将该建议应用到当前选定段落
   **And** 不会自动替换整篇文档或影响其他未选中段落
2. **Given** 用户查看某个段落的建议比较器
   **When** 用户对某个建议执行拒绝
   **Then** 系统记录该建议已被拒绝
   **And** 当前段落仍保留原文或已确认的其他修改结果
3. **Given** 当前段落存在多个候选建议
   **When** 用户需要做出决策
   **Then** 页面通过“修改决策条”提供采纳、拒绝和继续处理等高频动作入口
   **And** 同一区域只保留一个最强主操作，危险操作不与普通操作同权呈现
4. **Given** 用户已对当前段落执行采纳或拒绝
   **When** 系统写入处理结果
   **Then** 系统为该段落记录明确的处理状态
   **And** 用户可以在工作台中识别该段落已被接受、被拒绝或仍保留原文
5. **Given** 用户连续处理多个高风险段落
   **When** 用户完成当前段落决策后继续下一段
   **Then** 工作台保持上下文连续，进入下一个待处理高风险段落
   **And** 不要求用户重新定位已处理内容或重新加载整个页面
6. **Given** 采纳或拒绝操作需要落盘
   **When** 系统写入修改决策
   **Then** 后端以 append-only 的 `edit_operations` 思路记录本次决策与作用范围
   **And** 为后续回看、撤销、导出筛选和排障保留可追踪依据
7. **Given** 用户通过键盘或辅助技术操作决策区
   **When** 用户聚焦采纳、拒绝或继续下一段等动作
   **Then** 这些控件均支持键盘激活、清晰焦点状态和可读操作说明
   **And** 满足 WCAG 2.1 AA 与关键操作可理解性要求

## Dev Notes

### Story Intent & Boundaries

- Story 3.2 的重点是把 Story 3.1 的“可比较建议”推进到“可执行决策”，但仍限定在单段范围内。
- Story 3.2 负责采纳、拒绝与当前段落处理状态回写；不负责完整修改历史摘要、忽略/误判反馈、局部撤销或一键优化。
- PDF 继续保持只读边界，不允许 edit decision 写入。

### Previous Story Intelligence

- Story 3.1 已完成 `rewrite_options` 边界、建议比较器与 PDF 只读说明。
- 当前工作台已经具备风险列表、诊断面板、建议比较器和 `selectedParagraphId` 切换能力。
- `risk_findings.handlingStatus` 继续承担工作台可见状态展示职责，因此 Story 3.2 要在决策落盘后同步更新该状态。

### Technical Requirements

- 服务端事实状态继续放在 TanStack Query，本地跨面板交互状态继续放在 Zustand。
- 决策写入必须遵守 append-only `edit_operations` 思路，不能通过覆盖原文事实记录模拟“已采纳”。
- 采纳 / 拒绝接口必须校验 `paragraphId`、`optionId` 与当前任务归属关系。
- 当前故事只实现最小可行的决策写入结构，为 Story 3.3 历史回看和 Story 3.5 撤销保留轨迹。

### Completion Notes

- 已接通 append-only `edit_operations` 持久化链路，并提供 `POST /api/tasks/[taskId]/edit-operations`。
- 采纳 / 拒绝只作用于当前段落，同时同步更新 `risk_findings.handlingStatus`。
- 比较器已支持选择候选方案，并通过 `TaskDecisionBar` 执行采纳、拒绝和继续下一段。
- PDF 继续保持只读边界，明确阻止 edit decision 写入。
- 已通过 `npm test`、`npm run lint`、`npm run build`。
- Code review 结论：无阻塞 findings。非阻塞风险：工作台文案存在中英混合，后续需统一清理。
