# Story 3.3: 修改历史、处理状态与已处理段落回看

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 用户,
I want 查看当前任务中各段落的处理状态、修改历史，并能重新回看之前处理过的高风险段落,
so that 我可以清楚掌握哪些内容已经处理、如何处理过，以及是否需要再次检查。

## Acceptance Criteria

1. **Given** 用户已对一个或多个高风险段落执行过采纳、拒绝或保留原文等决策
   **When** 用户查看任务中的历史与状态摘要区域
   **Then** 系统展示各段落的处理状态汇总
   **And** 用户能够识别哪些段落已被接受、已被拒绝、已被忽略或仍未处理
2. **Given** 用户需要回看之前处理过的段落
   **When** 用户从风险列表、状态摘要或历史入口中选择某个已处理段落
   **Then** 系统重新打开该段落的诊断信息、原文与已记录决策
   **And** 用户无需重新触发扫描或重新生成整个任务结果
3. **Given** 某个段落已存在一系列处理动作
   **When** 用户查看该段落的修改历史
   **Then** 系统展示与该段落相关的关键决策轨迹和最近状态
   **And** 这些信息基于可追踪的 `edit_operations` 记录生成
4. **Given** 用户在多个已处理与未处理段落之间切换
   **When** 工作台更新当前选中项
   **Then** 当前段落状态、历史摘要和建议区域同步更新
   **And** 常规切换过程保持连续，不出现状态错位或明显阻塞
5. **Given** 当前任务尚未有任何已处理段落或某类状态暂无记录
   **When** 历史与状态摘要区域渲染
   **Then** 系统展示清晰空态说明
   **And** 告知用户接下来可执行的处理动作
6. **Given** 用户通过键盘或辅助技术访问历史与状态摘要组件
   **When** 用户在状态标签、历史记录和跳转入口之间导航
   **Then** 所有信息均具备可读文本、合理语义结构和清晰焦点顺序
   **And** 满足 WCAG 2.1 AA，且状态不只依赖颜色表达

## Dev Notes

### Story Intent & Boundaries

- Story 3.3 负责“看得到处理状态”和“能回看已处理段落”，不引入忽略/误判反馈或局部撤销能力。
- 仍沿用 Story 3.2 的 append-only `edit_operations` 事实源，不把历史摘要复制到 Zustand。
- 当前故事只做工作台历史摘要和段落回看，不实现导出前检查或一键优化入口。

### Previous Story Intelligence

- Story 3.2 已建立 `edit_operations` append-only 决策落盘，并提供 `POST /api/tasks/[taskId]/edit-operations`。
- 工作台已经具备风险列表、诊断面板、建议比较器、决策条和连续切段能力。
- `risk_findings.handlingStatus` 已扩展为 `pending / accepted / rejected / ignored`，可直接用于状态汇总。

### Technical Requirements

- 服务端事实状态继续由 TanStack Query 承载；Zustand 只保留当前选中段落等交互状态。
- 历史摘要以 `risk_findings` 汇总全量状态，以 `edit_operations` 提供关键轨迹与最近操作信息。
- `GET /api/tasks/[taskId]/edit-operations` 允许通过 `paragraphId` 读取当前段落历史，但不暴露跨任务数据。
- 历史面板必须与当前工作台切段联动，同步刷新而不造成状态错位。
