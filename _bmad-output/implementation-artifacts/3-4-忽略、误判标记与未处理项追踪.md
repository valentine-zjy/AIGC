# Story 3.4: 忽略、误判标记与未处理项追踪

Status: done

## Story

As a 用户,
I want 对段落或建议执行忽略、标记疑似误判或无帮助，并查看仍未处理的高风险项,
so that 我可以聚焦真正值得修改的内容，并在导出前知道还有哪些风险尚未决策。

## Acceptance Criteria

1. **Given** 用户正在查看一个高风险段落或某条候选建议
   **When** 用户选择忽略该段落或该建议
   **Then** 系统记录该对象已被忽略
   **And** 该状态在风险列表、状态摘要和后续导出前检查中可见
2. **Given** 用户认为某个段落或建议存在问题
   **When** 用户标记其为疑似误判、不认同或无帮助
   **Then** 系统记录对应反馈类型和目标对象
   **And** 当前任务继续可用，不因该反馈阻断后续处理流程
3. **Given** 用户需要查看仍未处理的风险项
   **When** 用户打开未处理项视图或使用筛选栏
   **Then** 系统展示当前仍未采纳、未拒绝、未忽略的高风险段落
   **And** 用户可以据此继续逐段处理
4. **Given** 用户已忽略或标记了部分建议和段落
   **When** 风险列表和状态摘要更新
   **Then** 系统将这些状态以可理解方式区分展示
   **And** 不仅依赖颜色，还提供可读文本或图形信号
5. **Given** 当前没有未处理项、没有可忽略对象或没有误判标记记录
   **When** 对应区域渲染
   **Then** 系统展示清晰的空态说明
   **And** 提供下一步指引，例如继续回看、准备导出或调整筛选条件
6. **Given** 用户通过键盘或辅助技术操作忽略与反馈组件
   **When** 用户聚焦忽略、误判标记和未处理项入口
   **Then** 所有操作都支持键盘激活、可见焦点和屏幕阅读器可理解文本
   **And** 满足 WCAG 2.1 AA 和关键状态说明可读性要求

## Dev Notes

- 3.4 没有新建第二套状态系统，而是让 feedback API 继续写 append-only `edit_operations`，再把对应段落状态回写为 `ignored`。
- 这样已有的风险列表、状态摘要、pending 筛选与历史面板都能直接复用。
- 反馈目标通过 `optionId` 和 operation type 记录，可追踪到当前建议。
