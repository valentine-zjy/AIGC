# Story 3.5: 局部撤销与可恢复编辑状态

Status: done

## Story

As a 用户,
I want 撤销单个局部修改并保持已确认编辑状态可恢复,
so that 我可以在某次局部改写不符合预期时安全回退，而不丢失其他已确认处理结果。

## Acceptance Criteria

1. **Given** 用户已对某个段落应用过局部修改
   **When** 用户对该段落执行撤销
   **Then** 系统仅回退该段落最近一次可撤销的局部修改
   **And** 不影响其他段落已确认的修改结果
2. **Given** 用户准备执行局部撤销
   **When** 系统展示回退确认界面
   **Then** 页面明确说明本次撤销的作用范围、可恢复性和主要后果
   **And** 这些说明出现在确认按钮附近
3. **Given** 局部撤销已执行成功
   **When** 系统更新当前段落内容与状态
   **Then** 该段落恢复到撤销前一版本的可继续编辑状态
   **And** 用户能够继续查看建议、重新决策或继续处理其他段落
4. **Given** 用户已确认过多个段落的修改
   **When** 系统发生处理中断、刷新或局部回退
   **Then** 已确认的其他局部修改不得无提示丢失
   **And** 系统优先保障这些修改具有可恢复性或可识别性
5. **Given** 系统需要记录本次局部撤销
   **When** 回退操作落盘
   **Then** 系统以 append-only 的 `edit_operations` 方式记录回退动作、作用对象和前后状态
   **And** 为后续历史查看、导出筛选和问题排查保留追踪依据
6. **Given** 用户通过键盘或辅助技术执行局部撤销
   **When** 用户操作回退确认组件
   **Then** 所有关键动作具备清晰焦点、可读文本和键盘可操作性
   **And** 满足 WCAG 2.1 AA 与危险操作提示要求

## Dev Notes

- 3.5 采用“最近一次已采纳改写可撤销”的最小可行模型：仅当段落最新操作为 `accept_suggestion` 时允许 rollback。
- 回退动作写入新的 `rollback_edit` operation，并把当前段落状态恢复到 `pending`，从而让建议重新可用。
- 其他段落的已采纳、已拒绝、已忽略状态完全不受影响。
