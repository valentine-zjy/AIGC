# Product Requirements Document - AI降重工具

**Author:** Valentin
**Date:** 2026-03-21

## Executive Summary

AI降重工具是一个面向正式长文档交付场景的编辑工作台型 Web 应用，首发切入点是毕业论文定稿前 48 小时的大学生用户。它解决的不是“写不出来”问题，而是“AI 已经帮我生成了草稿，但这份稿子我还不敢正式提交”这个最后一公里问题。用户已经广泛使用大模型完成研究、整理、扩写和初稿生成，但在正式提交前，仍要投入大量人工时间处理模板化表达、作者立场不清、论证空泛、句式节奏过于整齐以及整体“像机器写的”风险。

现有替代方案主要有四类：用户自己逐句手改、反复更换 prompt 让模型重写、叠加多个润色工具来回折腾、以及寻找真人润色。它们的共同问题不是完全无效，而是不稳定、不可控、耗时，并且很难在正式场景里建立“我敢交”的信心。AI降重工具的目标不是再做一个黑箱重写器，而是把最后一轮高价值编辑流程产品化：上传 Word、PDF、TXT 等长文档后，系统先定位高风险段落，解释为什么这些内容读起来像模板或不像真实作者写法，再提供 2-3 种局部修改建议，并支持带护栏的一键优化模式。

这个产品的核心价值排序非常明确。第一价值是“更敢交”，即让用户在保留原意的前提下，把稿件修到自己敢正式提交。第二价值是“更可控”，即整个修改过程不是黑箱整篇重写，而是解释先行、局部修改优先、支持对照与回退。速度很重要，但它是结果，不是主卖点。首发市场按 edtech 切入，但产品本质不是单一教育工具，而是一个可向更广泛长文档写作、编辑和知识工作场景扩展的后编辑工作台。

### What Makes This Special

这个产品真正不同的地方，不是“能把文章改得更顺”，而是“能把用户不敢交的稿子修到敢交”。它与通用改写器的差异不在模型本身，而在工作流设计：先诊断问题，再解释问题，最后才进入局部精修或一键处理。它不默认整篇重写，而是优先处理最值得修改的高风险部分；不要求用户盲目信任输出，而是通过原因解释、原文对照、可选方案和回退能力建立信任。

核心洞察是：今天用户的瓶颈已经不是生成初稿，而是如何在正式交付前完成一轮可控、可信、可解释的后编辑。谁能把这条低效、混乱、依赖多工具和人工兜底的链路压缩掉，谁就不是在卖“润色”，而是在卖“交付信心”。这也是为什么用户会选择它而不是现有替代方案：它不是让用户换一种折腾方式，而是替代那条折腾链路本身。

## Project Classification

- Project Type: Web application，具体形态为编辑工作台型文档处理与局部优化产品
- Domain: EdTech 切入，首发核心用户为毕业论文定稿前 48 小时的大学生，后续可扩展到更广泛写作与编辑场景
- Complexity: Medium，复杂度主要来自长文档处理、多格式导入导出、解释式编辑、用户信任与隐私要求
- Project Context: Greenfield

## Success Criteria

### User Success

用户成功的定义不是“文档被改写过”，而是“用户在正式交付前，对稿件质量和可提交性建立了明显信心”。对首发用户而言，成功发生在三个连续时刻：用户上传文档后，系统能快速指出最值得优先处理的高风险段落；用户读完解释后认同这些问题确实存在；用户采纳局部建议或护栏式一键优化后，感受到稿件更像自己会提交的版本，而不是另一篇失控重写稿。

用户层面的完成状态应满足以下条件：用户能定位 Top 风险段落、理解每个问题为何存在、完成至少一轮局部优化或一键优化，并顺利导出结果。真正的 aha moment 不是“工具改得很快”，而是“它准确找到了我最担心的地方，而且没有把整篇稿子改跑偏”。

### Business Success

业务成功分为验证期和扩张期。验证期的核心不是规模，而是证明这个产品确实替代了用户当前那条低效的修改链路；因此最关键的不是注册数，而是有多少文档完成了有效优化，有多少用户在第一次使用后愿意继续用第二篇文档。扩张期才开始看更强的复用、推荐和更广泛用户群扩展。

业务上应优先关注三类信号：第一，产品是否真正解决高压正式交付场景里的“最后一轮编辑”问题；第二，用户是否愿意从单次应急使用转向持续复用；第三，是否开始出现口碑驱动的新用户增长，而不只是一次性流量获取。

### Technical Success

技术成功的标准不是抽象的模型能力，而是对正式文档场景的可靠支撑。系统必须稳定处理长文档和多格式文件，扫描结果要足够快，局部建议要能和原文保持清晰映射，修改过程要支持对照与回退，导出文档要可直接继续编辑或提交。对这个产品来说，任何数据丢失、格式损坏、段落错位或不可解释的修改，都会直接破坏信任。

因此，技术成功至少包含四个层面：长文档处理稳定性、多格式导入导出可用性、编辑过程可追踪可回退、以及隐私和数据处理边界清晰。这里的重点是“交付级稳定”，不是 demo 级可运行。

### Measurable Outcomes

北极星指标建议定义为：**有效优化文档占比**。即在完成一次扫描和修改流程后，被用户认为关键问题得到实质改善的文档占比。这个指标最贴近产品的真实价值，因为它同时反映识别质量、建议质量和最终交付信心。

配套指标建议包括：
- 文档流程完成率：上传后完成扫描、修改、导出的文档占比
- 首次扫描完成时间：用户上传到看到首轮结果的时间
- Top 问题段落查看率：用户是否真的进入高风险段落列表
- 首次采纳建议时间：从看到结果到第一次接受建议的耗时
- 每篇文档被采纳的优化段落数：衡量建议是否真正被使用
- 建议采纳率：衡量建议质量与可信度
- 一键优化使用率：衡量高压场景下的效率需求
- 编辑工作台使用率：衡量解释式、可控式编辑路径是否成立
- 用户满意度评分：衡量主观质量感知
- 推荐来源新用户占比：衡量口碑是否开始形成
- 复用率：用户是否把第二篇、第三篇文档继续交给系统处理

## Product Scope

### MVP - Minimum Viable Product

MVP 只覆盖能证明“更敢交 + 更可控”的最短闭环，不做泛化写作平台。范围包括：支持 Word、PDF、TXT 上传；完成扫描并输出 Top 高风险段落；解释问题原因；提供 2-3 种局部修改建议；支持采纳、对照和回退；提供带护栏的一键优化模式；支持导出修改结果。

PDF 在 MVP 中只承诺读取与扫描，不承诺复杂版式下的精确回写。任何不能直接提升交付信心或可控性的能力，都不属于 MVP。

### Growth Features (Post-MVP)

Post-MVP 的增长能力围绕“更强复用”和“更强竞争力”展开，重点包括作者声音保持与风格偏好、跨文档复用、更稳定的长文档与批量处理、版本历史、修改报告、更多格式支持，以及面向不同写作场景的模式化能力。

### Vision (Future)

长期愿景不是做一个更大的改写器，而是成为正式长文档交付前的统一后编辑层，逐步扩展到更广泛的文字工作者、编辑、研究支持和团队协作场景。最终理想状态下，用户不再把它理解为“润色工具”，而是理解为“正式交付前最后一定会过一遍的质量工作台”。

## User Journeys

### Journey 1: 主用户成功路径 - 毕业论文定稿前 48 小时的学生

李晴，22 岁，正在准备提交毕业论文，距离学院要求的最终提交时间只剩 36 小时。论文主体已经完成，大量段落来自她过去几周反复使用大模型辅助生成、扩写和整理后的版本。现在最困扰她的不是“内容不够”，而是她不敢交。她知道整篇论文逻辑基本通顺，但有些段落读起来很像模板，论证太平，像在复述标准答案，而不是她自己写出来的正式成稿。

她通过朋友推荐进入产品，上传 Word 文档，选择“编辑工作台模式”。系统在较短时间内完成扫描，并直接给出 Top 高风险段落列表。李晴第一眼看到的不是整篇重写结果，而是几个明确的问题点：某段“立场不清”、某段“句式节奏过于整齐”、某段“论证空泛但结论先行”。她点开其中一段，看到原因解释、原文对照，以及 2-3 种不同方向的局部修改建议。她开始逐段采纳，保留自己认可的表达，拒绝不合意的建议，并在必要时回退。

真正的高潮出现在她处理完最担心的几段后，重新通读文档，发现稿件仍然是“她的论文”，但已经不再像被批量生成的草稿。她导出修改结果，开始产生“这稿子我敢交了”的感觉。她的情绪从焦虑和不确定，转向可控和安心。

**这条旅程揭示的能力需求：**
- 多格式文档上传与稳定扫描
- Top 风险段落识别与排序
- 问题原因解释层
- 2-3 种局部建议
- 原文对照、采纳、拒绝、回退
- 导出可继续提交的文档

### Journey 2: 主用户边界路径 - 结果不完全信服时的恢复过程

同样是李晴，但这次她上传的是一个格式复杂的 PDF 版本。系统完成扫描后，给出一些高风险段落判断，但她对其中两条并不认同。她觉得某些段落虽然风格整齐，但那本来就是她在学术场景里故意保持的正式表达。如果系统在这里一味推动改写，她会迅速失去信任，甚至直接放弃工具。

她点开争议段落，希望先看到“为什么系统这样判断”，而不是直接被迫接受修改结果。系统展示具体原因，并清楚地区分“高置信建议”和“可选建议”。她发现其中一条建议确实有帮助，另一条则选择忽略。接着，她在一个段落里误用了自动优化，发现改写偏离原意，于是立即回退到修改前版本，再改用局部建议。

这条旅程的关键不在顺畅，而在恢复。用户不是要求系统永远判断正确，而是要求系统在判断不完全准确时，仍然给她保留控制权、可解释性和撤销路径。只要恢复路径清楚，她就不会因为一次分歧而彻底失去信任。

**这条旅程揭示的能力需求：**
- 建议置信度或强弱提示
- 解释先于修改
- 忽略建议能力
- 一键优化的可回退机制
- 修改历史和版本恢复
- PDF 场景下尽量稳定的段落映射

### Journey 3: 次用户路径 - 长文写作者/编辑的持续复用

周岚，31 岁，是一名长文写作者兼内容编辑。她不是在论文冲刺，而是在完成一篇需要对外发布的长文稿件。她并不一定因为时间极度紧张才使用产品，而是因为她已经形成了一种工作方式：先用 AI 帮助整理框架和初稿，再进入精修阶段。但她不接受整篇被黑箱重写，因为那会破坏作者声音，也会让她重新审核大量内容。

她上传文稿后，更看重的是“哪些地方最不像我”，而不是“哪里不够通顺”。她逐步发现这个工具不是普通润色器，因为它会把问题类型说清楚，例如“表达空泛”“作者判断缺位”“句式节奏过于均匀”。她选择其中一种更接近自己风格的局部建议，逐段修整。随着使用次数增加，她开始把这个产品视为正式交付前的固定环节，而不是一次性应急工具。

她的成功不是“第一次终于能用了”，而是“第二篇、第三篇稿子我还会继续回来”。这意味着产品不只解决一次性焦虑，还形成了稳定复用。

**这条旅程揭示的能力需求：**
- 支持长文持续编辑的工作台体验
- 问题类型分类而非仅输出改写
- 更强调风格保持和作者声音
- 复用体验稳定
- 导出结果适合继续人工编辑
- 为后续风格档位或偏好学习留空间

### Journey 4: 内部运营支持路径 - 单人创始人处理异常与质量问题

这个角色不是独立团队，而是你自己。你既是产品负责人，也是运营支持和异常处理者。某天，一个用户上传文档后扫描失败，或者导出结果格式错乱，或者质疑某个高风险判断明显不合理。如果系统没有给你最基本的任务状态、失败原因、文档处理日志和问题定位入口，你就无法判断问题出在上传、解析、扫描还是导出环节。

你进入内部后台或最小运营面板，查看最近任务状态，发现某个 PDF 因结构异常导致段落切分错误；另一个任务因为导出阶段失败而中断。你需要能快速重试任务、查看失败原因、定位高频问题类型，并在必要时调整基础规则或屏蔽某类不稳定处理路径。你不是在“运营一个团队”，而是在“确保产品还能被信任地继续使用”。

这条旅程的价值在于，它决定了产品能不能从 demo 变成可运营产品。只要没有最基本的内部排障和质量追踪能力，用户端再好的体验也无法稳定维持。

**这条旅程揭示的能力需求：**
- 基础任务状态可视化
- 上传、解析、扫描、导出各阶段状态追踪
- 错误日志与失败原因
- 失败任务重试
- 问题样本回看与质量校验入口
- 基础规则或策略调整能力

### Journey Requirements Summary

这些旅程共同揭示出几组核心能力：

1. 用户侧核心能力必须围绕“更敢交”和“更可控”展开，而不是围绕泛化改写展开。系统需要先定位高风险段落、解释问题、再支持局部建议和可回退的一键模式。
2. 旅程明确要求产品同时支持两种使用方式：工作台式精修和高压场景下一键优化，但后一种必须有护栏和撤销能力。
3. 次用户旅程说明产品不能只服务一次性学生场景，还必须具备持续复用的编辑工作台特征。
4. 内部运营支持旅程说明 MVP 不能只有用户前台，也必须至少具备最小可用的任务监控、错误定位和失败恢复能力。
5. 由于暂不开放 API，当前功能边界应集中在文档上传、扫描、局部编辑、导出和内部质量运营，不引入外部集成复杂度。

## Domain-Specific Requirements

### Compliance & Regulatory

本产品虽然不属于强监管行业，但处于教育场景，且处理的是学生正式文档，因此必须满足基础教育隐私与可访问性要求。MVP 阶段不按 K-12 设计，不把 COPPA 作为核心前提，但应默认以高教场景处理学生文档数据，并为未来可能的院校合作保留 FERPA 相关约束的兼容空间。系统还应满足基础无障碍要求，至少以 WCAG 2.1 AA 作为 Web 端交互设计的目标基线。

另一个关键边界不是法规条文，而是学术诚信定位。产品不得被定义为代写、规避审核或绕过学校规范的工具，而应明确定位为写作质量诊断、表达优化和编辑辅助产品。所有对外文案、用户引导和产品内说明，都应避免暗示“规避 AI 检测”这类用途。

### Technical Constraints

技术约束的重点是文档隐私、处理稳定性和解释可信度。用户上传的往往是尚未提交的论文、作业或正式稿件，因此系统需要清楚界定文档是否存储、存储多久、谁可访问、是否用于训练、用户如何删除。即使早期是单人产品，也不能在这部分含糊。

第二个约束是长文档与多格式稳定性。高教用户文档通常较长，且 Word、PDF 的结构差异大，系统必须优先保证段落切分、问题定位和导出质量，而不是追求过度复杂的自动化。第三个约束是建议的可解释性。学术写作里并非所有“整齐”“正式”的句子都应被视为问题，因此系统应尽量区分高置信风险与可选优化，避免把正常学术表达误判为模板化内容。

### Integration Requirements

MVP 阶段不引入开放 API，也不以学校系统集成为前提。当前最核心的“集成”其实是文档输入输出链路本身：稳定支持 Word、PDF、TXT 上传，稳定输出可继续编辑或提交的结果文档。这一点在你的场景里比 LMS、SSO、插件集成更重要。

后续若进入增长阶段，可以评估与 Word 插件、云文档平台、学校写作中心或机构采购流程的适配，但这些都不应反过来定义 MVP。当前阶段的集成要求应收敛为：文件导入、文件导出、任务状态追踪和最小可用的内部运营排障能力。

### Risk Mitigations

这个领域最突出的风险有四类。第一类是定位风险：如果产品被用户或外界理解为规避 AI 审核的工具，会直接损害品牌、传播路径和未来机构接受度。缓解方式是从 PRD 开始明确产品边界，统一对外表达为“质量诊断与编辑辅助”。

第二类是隐私风险：学生可能上传尚未提交的论文和正式材料，一旦数据处理边界不清、存储策略含糊或内部可见范围失控，信任会迅速崩塌。缓解方式是最早期就建立清晰的数据保留、删除、访问和训练边界策略。

第三类是误改风险：如果系统把本来合理的学术表达误判为模板化，或者在一键优化中改坏了原意，用户会很快失去“敢交”的信心。缓解方式是解释先行、局部修改优先、提供回退与原文对照，并尽量标注建议强弱。

第四类是格式与交付风险：用户最终要提交的是文档，不是网页结果。如果导出后的结构、段落、标点或格式出问题，前面的优化价值会被直接抵消。缓解方式是把导出质量视为 MVP 核心能力，而不是边缘收尾功能。

## Innovation & Novel Patterns

### Detected Innovation Areas

本产品的创新不在模型层，而在正式交付前的人机协作编辑流程。现有工具大多把重点放在生成、润色或整篇改写上；本产品重新组织的是“AI 草稿到正式成稿”之间最后一轮高价值编辑：先识别最值得修改的高风险段落，再解释为什么这些内容读起来像模板、像机器或不像真实作者写法，最后才进入局部精修或护栏式一键优化。

这种创新首先体现在交互顺序上。用户第一次感受到产品不同，不是因为看见了一个更强的改写框，而是因为第一屏出现的是 Top 高风险段落清单，而不是整篇一键重写入口；第一动作是理解“为什么这里像模板”，而不是盲目接受改写；第一价值是“终于知道该先改哪”，而不是“又生成了一版”。本质上，这个产品不是让 AI 替用户重写，而是让 AI 成为一个可解释、可回退的最后一轮编辑台。

### Market Context & Competitive Landscape

当前市场已经有大量解决局部问题的工具，但缺少一个把“正式交付前最后一轮可控编辑”当成主任务来设计的产品。通用写作工具主要解决语法、流畅度和轻量改写；学术写作工具主要解决学术英语、投稿表达和论文语言规范；文档 AI 工具主要解决文档理解、问答和总结。你的产品切开的，是这些工具都没有完整接住的那条链路：当用户已经有一份可用初稿，但仍然不敢正式提交时，如何完成最后一轮高价值、可控、可信的修改。

因此，这里的创新不应表述为“从来没人做过 AI 编辑”，而应表述为：市场上已有很多生成、润色和文档处理工具，但还没有人把“AI 后编辑工作台”作为一个独立任务流和产品心智清楚定义出来。

### Validation Approach

这类创新最需要验证的不是模型评分，而是用户是否真的接受这种新工作流。验证应分三层进行。

第一，验证“风险段落优先”的交互是否成立。观察目标用户上传真实文档后，是否愿意优先查看 Top 问题段落，而不是直接寻找整篇一键重写入口。第二，验证“解释先行”是否提升信任。测试用户是否因为看到了问题原因、原文对照和建议强弱，而更愿意采纳建议。第三，验证“更敢交”是否是可感知价值。让用户在修改前后分别对“我敢不敢正式提交这份稿子”做主观评分，并结合建议采纳率、回退率和导出率判断是否真正建立了交付信心。

MVP 阶段不需要证明所有用户都喜欢这种交互，只需要证明在论文冲刺和长文正式交付场景里，这种工作流比“整篇重写”更值得信任。

### Risk Mitigation

这类创新也有明确风险。第一，用户口头上说想要可控编辑，但行为上仍然优先寻找“一键全改”。缓解方式不是放弃工作台，而是保留护栏式一键模式作为效率分支，让两种路径在同一产品内共存。第二，用户可能认为“解释先于结果”增加了步骤。缓解方式是把解释做成高密度、可快速扫读、与建议紧邻的界面，而不是长篇说明。

第三，市场教育风险很高。如果“AI 后编辑工作台”这个概念讲不清，用户会继续把产品误解成普通润色器。缓解方式是坚持围绕真实任务来讲产品，不讲抽象范式，而讲“这是你在正式提交前最后会用的一层质量工作台”。第四，如果这套交互在学生场景不成立，fallback 不是退回泛化写作平台，而是继续收缩到更明确的高压正式交付场景，把问题排序、解释和导出能力做得更强。

## Web Application Specific Requirements

### Project-Type Overview

本产品的核心交付形态确定为桌面优先的 SPA Web 应用，服务于长文档上传、风险段落扫描、局部编辑、对照采纳、回退和导出这一条连续工作流。之所以选择 SPA，而不是 MPA，是因为该产品的核心价值不在页面跳转效率，而在同一上下文内持续完成“看问题、理解原因、尝试修改、比较结果、决定采纳”的高频交互。对于毕业论文定稿前 48 小时的用户来说，保持状态连续性和编辑上下文稳定性，比传统多页结构更重要。

首发体验应明确采用桌面优先策略。主场景是用户处理 2 万字到 10 万字以上的正式文档，且需要并排查看原文、问题解释、修改建议和版本差异，这天然更适合桌面端完成。移动端在 MVP 中不承担完整编辑工作台职责，而应以基础可访问、可查看任务状态和可进行轻量操作为目标，而不是追求全量等价体验。

### Technical Architecture Considerations

前端架构应围绕长任务、长文本和强状态编辑来设计。由于不做多人实时协作，系统不需要为 MVP 引入复杂的协同编辑或实时同步架构；更合适的方案是以异步任务处理为核心，通过稳定的任务状态更新机制支撑上传、解析、扫描、建议生成和导出流程。对用户而言，关键不是“是否实时协作”，而是“系统是否持续给出清晰、可信的进度反馈”。

SPA 架构还要求前端具备较强的状态管理能力，以支持以下关键交互：高风险段落列表与正文定位映射、原文与建议对照、采纳与拒绝操作、局部回退、一键优化后的撤销，以及导出前的最终确认。这里的技术重点不是页面数量，而是单一工作台内多面板、多状态、多版本切换的一致性和可恢复性。

### Browser Matrix

MVP 的一线支持环境应定义为桌面端现代 Chromium 浏览器，优先支持 Windows 环境下的 Chrome 与 Edge。原因是首发主用户群体以高校学生为主，实际使用环境大概率集中在 Windows 笔记本与实验室电脑，Chrome / Edge 也是最常见的正式写作工具链浏览器。

建议支持基线定义为：
- Windows 10 / 11 上的最新版及前一主要版本 Chrome
- Windows 10 / 11 上的最新版及前一主要版本 Edge
- macOS 最近两个主版本上的 Chrome 作为可用支持

对于 Firefox 与 Safari，MVP 阶段可定义为“尽量兼容但不作为首发承诺环境”。这样可以避免团队在首阶段把验证资源消耗在低优先级浏览器适配上，同时保持未来扩展空间。

### Responsive Design

响应式策略应明确采用“桌面优先，移动兜底”。桌面端需要完整支持工作台核心布局，包括风险段落列表、正文定位区、原因解释区、建议区、对照预览区和操作区。对于主要目标用户而言，桌面体验不是增强项，而是主产品形态。

平板和移动端在 MVP 中应支持以下最小可用能力：
- 登录与进入任务
- 上传文档
- 查看扫描状态与 Top 风险段落
- 查看单段解释与建议
- 进行有限的采纳或标记操作

移动端不应强行承诺完整的长文档精修体验，尤其不应把复杂对照编辑和大篇幅版本管理作为首发目标。这样的边界有助于产品聚焦，也更符合“更敢交、更可控”的核心任务。

### Performance Targets

性能目标应围绕“尽快进入可判断、可行动状态”来设定，而不是单纯追求绝对最低延迟。对这个产品来说，真正影响体验的不是每一步都极快，而是用户能否快速看到系统已经接管任务、知道现在进行到哪一步、以及多久能拿到第一批高价值结果。

MVP 建议采用如下性能目标：
- 工作台首屏在目标桌面环境中达到 2 秒内可交互
- 文档上传后 3 秒内给出明确的任务受理与状态反馈
- 常见论文长度文档（约 2 万到 5 万字）在 90 秒内返回第一版 Top 风险段落结果
- 更长文档（约 5 万到 10 万字及以上）采用异步处理，并持续展示进度，目标是在 5 分钟内返回首轮可操作结果
- 用户确认导出后，应在可接受时间内生成结果文件，并向用户明确展示导出成功或失败原因

这里的性能原则应是“渐进反馈优先于静默等待，稳定可恢复优先于极限提速”。

### SEO Strategy

SEO 不应作为核心工作台的产品目标。真正需要 SEO 的是官网、介绍页、落地页、帮助中心和公开说明页面，而不是登录后的编辑工作台本身。因为用户进入核心产品后的目标是上传、扫描、修改和导出，不是被搜索引擎索引。

因此，产品应采用分层 SEO 策略：
- 公开官网与营销页面承担搜索获客与概念教育
- 登录后的工作台区域默认不作为 SEO 目标
- 与具体用户文档、任务结果、编辑状态相关的页面全部禁止被索引

这也与产品定位一致：核心价值来自可信的后编辑工作流，而不是公域内容曝光。

### Accessibility Level

Web 端无障碍基线应明确采用 WCAG 2.1 AA。对于该产品，这不是“锦上添花”，而是正式写作与教育场景下的基础可信要求。由于核心工作流涉及长文阅读、比对修改、状态判断和操作确认，无障碍设计应从 MVP 开始纳入，而不是后补。

MVP 至少应满足以下无障碍要求：
- 全流程可键盘操作
- 清晰可见的焦点状态
- 足够的文本与背景对比度
- 表单、按钮、状态提示和操作反馈具备明确语义标签
- 风险等级、建议强弱和修改状态不能只依赖颜色传达
- 重要对照视图与任务状态对屏幕阅读器可理解
- 错误提示、上传失败、导出失败等关键状态应具备明确可读反馈

考虑到用户可能在高压、疲劳和长时间阅读环境下使用产品，无障碍不仅服务特定人群，也直接提升整体可用性和信任感。

### Implementation Considerations

实现层面应优先保障三个原则：第一，桌面工作台体验必须稳定；第二，长任务进度反馈必须清楚；第三，编辑行为必须可恢复。与其在 MVP 阶段追求跨浏览器全量一致、移动端全功能覆盖或复杂实时协作，不如把资源集中在上传解析稳定性、风险段落定位准确性、局部修改映射、撤销回退和导出质量上。

技术落地上，前端应为后续扩展预留空间，例如未来增加更强的任务通知、更丰富的浏览器支持或更细的响应式能力，但 MVP 阶段不应因这些扩展性预设而显著增加系统复杂度。这个阶段的成功标准不是“像一个完整平台”，而是“像一个可以被正式使用和信任的长文档后编辑工作台”。

## Project Scoping & Phased Development

### MVP Strategy & Philosophy

**MVP Approach:** 问题解决型 MVP  
**Resource Requirements:** 以单人创始人为核心推进的极小可行开发模式。MVP 阶段默认由同一人承担产品、前端、后端、AI 编排、文档处理与基础运维职责，因此范围必须围绕“最少功能证明核心价值”来收敛，而不能按完整平台思路展开。

这个 MVP 的目标不是证明平台规模化能力，也不是优先验证商业化闭环，而是尽快验证一个清晰命题：用户在正式提交前，是否会因为这套工作流而明显感到“更敢交”。因此，MVP 应优先服务高压正式交付场景下的最后一轮后编辑，而不是扩展到泛写作、多人协作或机构集成。

MVP 的哲学应明确为：优先解决一个尖锐问题，而不是展示一套完整能力。只要用户上传文档后，系统能明确指出最值得处理的问题段落、解释原因、给出局部可采纳建议，并让用户在可控前提下导出更敢提交的版本，这个 MVP 就成立。

### MVP Feature Set (Phase 1)

**Core User Journeys Supported:**
- 毕业论文定稿前 48 小时学生用户的核心成功路径
- 对部分判断不完全认同时的恢复路径
- 单人运营者最小任务监控与失败排查路径

**Must-Have Capabilities:**
- 支持 Word、PDF、TXT 文档上传
- 支持异步扫描任务，并清晰展示处理状态
- 输出 Top 高风险段落清单，而不是默认整篇重写
- 对每个高风险段落解释“为什么这里像模板 / AI 味重 / 不像真实作者写法”
- 为每个段落提供 2-3 种局部改写建议
- 支持采纳、拒绝、对照查看与回退
- 保留带护栏的一键优化模式，但不作为唯一主路径
- 支持导出修改结果，且结果应可继续编辑或继续提交
- 提供最小可用的内部任务日志、失败原因定位与重试能力

**MVP Boundary Decisions:**
- PDF 在 MVP 中先实现“读取与扫描”，不承诺复杂版式下的精确回写
- 桌面端为主战场，移动端只承担查看结果与轻操作，不做完整长文精修体验
- 不做多人实时协作
- 不做开放 API
- 不做学校系统、插件或外部平台集成
- 不做复杂后台系统，只做最小运营排障能力

### Post-MVP Features

**Phase 2 (Post-MVP):**
- 更强的导出质量与格式稳定性
- 更强的作者声音保持与风格偏好设置
- 批量处理更长文档或多文档
- 修改报告与优化摘要
- 更稳定的 PDF 映射和导出体验
- 更完整的移动端辅助体验

**Phase 3 (Expansion):**
- 跨文档风格记忆与连续性优化
- 团队协作与编辑协同流程
- 面向文字工作者、编辑和机构用户的扩展版本
- 更强的组织级工作流和权限能力
- 向更广泛正式写作与内容生产场景扩张

### Risk Mitigation Strategy

**Technical Risks:**  
最高技术风险是长文档处理稳定性、PDF 段落映射、局部修改后的一致性，以及导出质量。缓解策略是首阶段优先保证 Word/TXT 的完整链路，把 PDF 收敛为“可扫描、可定位、弱回写承诺”的输入渠道；同时把回退、对照和任务状态反馈作为核心能力，而不是边缘补丁。

**Market Risks:**  
最大市场风险不是“用户需不需要润色”，而是用户是否真的会把这套产品当作正式提交前的最后一道工序。另一个风险是被误解为规避 AI 检测工具。缓解策略是坚持“质量诊断与后编辑工作台”的定位，并把 MVP 的首要验证信号锁定为：用户看完结果后是否明显觉得自己更敢交。

**Resource Risks:**  
最大资源风险是单人开发下范围失控。缓解策略是严格冻结 MVP 边界，不同时追求多格式全保真、移动端全功能、实时协作、平台化扩展和商业化复杂能力。若资源进一步受限，优先保留上传、扫描、Top 风险段落、解释、局部建议、回退和导出这条最短闭环。

## Functional Requirements

### Product Access & Task Initiation

- FR1: 访客可以在开始使用前查看产品用途、适用场景和使用边界说明。
- FR2: 用户可以在开始任务前了解支持的文档格式及对应的处理边界。
- FR3: 用户可以在不经过复杂账户流程的前提下启动一次文档处理任务。
- FR4: 用户可以在扫描开始前选择工作台模式或一键优化模式。
- FR5: 用户可以上传 Word、PDF 或 TXT 文档进行处理。
- FR6: 用户可以看到任务处于已受理、处理中、已完成或失败等状态。
- FR7: 用户可以在任务失败时看到明确的失败原因和下一步处理提示。

### Content Diagnosis & Risk Prioritization

- FR8: 用户可以收到文档 Top 高风险段落的排序结果。
- FR9: 用户可以在原文上下文中查看被标记的高风险段落。
- FR10: 用户可以查看每个高风险段落对应的问题类型。
- FR11: 用户可以理解每个高风险段落为何被判定为模板化、AI 味重或不像真实作者写法。
- FR12: 用户可以区分高置信问题与可选优化项。
- FR13: 用户可以围绕高风险段落进行优先处理，而无需先整篇重写。

### Local Rewrite Suggestions

- FR14: 用户可以查看每个高风险段落的 2-3 种局部改写建议。
- FR15: 用户可以将局部改写建议与原文进行对照查看。
- FR16: 用户可以对单个段落选择并应用某个改写建议。
- FR17: 用户可以只对选定段落应用修改，而不替换整篇文档。
- FR18: 用户可以在同一任务中重新回看之前处理过的高风险段落。

### Guardrailed Optimization & Editing Control

- FR19: 用户可以对符合条件的文档触发带护栏的一键优化流程。
- FR20: 用户可以在最终导出前查看一键优化后的结果。
- FR21: 用户可以在一键优化后继续进入局部精修流程。
- FR22: 用户可以按段落接受或拒绝建议修改。
- FR23: 用户可以撤销单个局部修改。
- FR24: 用户可以撤销一键优化带来的修改结果。

### Feedback, Resolution & Decision Support

- FR25: 用户可以在当前任务中忽略某个高风险段落或某条建议。
- FR26: 用户可以标记某个高风险段落或建议为不认同、无帮助或疑似误判。
- FR27: 用户可以查看当前任务中哪些修改已被接受、拒绝或保留原文。
- FR28: 用户可以在导出前查看仍未处理的高风险段落。
- FR29: 用户可以在查看结果后提交满意度或主观反馈。
- FR30: 用户可以反馈本次结果是否让文档变得更敢正式提交。

### Export, Data Handling & Document Lifecycle

- FR31: 用户可以将修改结果导出为可继续编辑的 Word 或 TXT 文档。
- FR32: 用户可以获得仅包含其已采纳修改的导出结果。
- FR33: 用户可以对 PDF 上传内容完成扫描与结果查看，即使暂不支持复杂版式下的精确回写。
- FR34: 用户可以了解不同输入格式在扫描、编辑和导出上的能力差异。
- FR35: 用户可以查看文档存储、访问、训练使用与删除规则说明。
- FR36: 用户可以请求删除当前上传文档及其相关任务数据。

### Internal Operations & Quality Recovery

- FR37: 系统运营者可以查看任务在上传、解析、扫描、优化和导出各阶段的状态。
- FR38: 系统运营者可以查看任务失败原因和基础处理记录。
- FR39: 系统运营者可以重试失败任务。
- FR40: 系统运营者可以在既定数据访问边界内查看排查质量问题所需的最小任务样本。
- FR41: 系统运营者可以识别重复出现的处理失败或质量投诉模式。
- FR42: 系统运营者可以在发现重复问题后调整或停用不稳定的处理规则或路径。

## Non-Functional Requirements

### Performance

- NFR1: 在目标桌面环境与受支持浏览器中，核心工作台首屏应在 2 秒内达到可交互状态。
- NFR2: 用户上传文档后，系统应在 3 秒内返回明确的任务受理反馈与初始状态。
- NFR3: 对于常见论文长度文档（约 2 万至 5 万字），系统应在 90 秒内返回首轮 Top 高风险段落结果。
- NFR4: 对于更长文档（约 5 万至 10 万字及以上），系统应在 5 分钟内返回首轮可操作结果，且在等待过程中持续提供可理解的任务状态反馈。
- NFR5: 在结果已生成的前提下，用户在工作台内切换风险段落、查看解释、查看建议与执行采纳或拒绝操作时，界面反馈应保持连续，不得因常规操作造成明显卡死或阻塞。
- NFR6: 当系统暂时无法满足目标处理时长时，必须优先保证任务状态可见、结果逐步可达和用户预期可管理，而不能进入无反馈等待状态。

### Security & Privacy

- NFR7: 用户上传文档及相关任务数据默认不得用于模型训练或未经声明的二次用途。
- NFR8: 用户文档内容、任务结果和导出结果在传输过程中必须加密保护。
- NFR9: 用户文档内容、任务结果和相关存储数据在静态存储时必须加密保护。
- NFR10: 只有用户本人及处于明确排障职责边界内的系统运营者可以访问任务相关数据，且该访问必须以最小必要原则为前提。
- NFR11: 系统必须向用户明确说明文档数据的存储范围、保留期限、访问边界、训练边界和删除方式。
- NFR12: 用户上传的文档及相关任务数据默认保留 7 天，超过保留期后系统应自动删除或不可恢复清除。
- NFR13: 用户发起删除请求后，系统应删除当前上传文档及其相关任务数据，并向用户提供明确的删除结果反馈。
- NFR14: 系统运营侧用于问题排查的任务样本访问必须受限于最小必要信息，不得将用户完整文档作为默认排障视图。

### Reliability

- NFR15: 系统不得以静默失败方式处理中断任务；任何失败都必须向用户或运营者显示明确状态。
- NFR16: 上传、解析、扫描、建议生成、优化和导出各阶段都必须具备可区分的任务状态，以支持定位失败环节。
- NFR17: 当任务失败时，系统必须向用户提供可理解的失败原因或失败类型，并给出下一步处理提示。
- NFR18: 用户已采纳的局部修改不得在无提示情况下丢失；如果发生处理中断，系统必须优先保障已确认修改的可恢复性或可识别性。
- NFR19: 当导出失败时，系统必须允许重试或重新发起导出，而不是要求用户重新完成整轮编辑流程。
- NFR20: 当一键优化结果不符合预期时，系统必须保证用户可以恢复到优化前的可继续编辑状态。
- NFR21: 系统应保留满足最小运营排障需求的任务处理记录，以支持重复故障识别与质量问题定位。

### Accessibility

- NFR22: Web 端体验应满足 WCAG 2.1 AA 作为无障碍基线。
- NFR23: 核心工作流，包括上传、模式选择、风险段落浏览、建议查看、采纳或拒绝、回退和导出，必须支持全键盘操作。
- NFR24: 所有可交互元素必须具备清晰可见的焦点状态，且焦点移动顺序应与任务流程一致。
- NFR25: 文本、状态提示和关键操作元素必须满足可读的颜色对比要求。
- NFR26: 风险等级、建议强弱、成功或失败状态和处理进度不得仅依赖颜色表达，必须提供额外可识别信号。
- NFR27: 关键状态变化，包括上传失败、扫描完成、导出失败和删除结果，必须对屏幕阅读器可理解。
- NFR28: 错误提示、帮助说明和关键边界说明必须以清晰文本形式呈现，不能仅以视觉布局暗示。

### Scalability

- NFR29: 系统应支持产品早期阶段的日常任务规模，能够稳定处理约 50 至 200 个文档任务的常规负载。
- NFR30: 在论文提交高峰期出现约 3 至 5 倍负载增长时，系统应优先保持任务受理、排队、状态反馈和结果可获取，而不是整体不可用。
- NFR31: 当系统容量逼近上限时，应优先采取可见的排队、延迟提示或降级处理，而不是无提示拒绝任务。
- NFR32: 扩展系统容量时，不应要求重写核心用户工作流或破坏既有任务处理边界。
